# 高级研发视角架构不足分析

评估日期：2026-05-13

## 结论

从 10 年经验高级研发工程师的角度看，当前项目已经具备一个可运行的 MVP 后端雏形：FastAPI 入口、服务层、LLM 解析、Playwright 执行、证据采集、报告生成和内存存储都已经串起来。

但当前架构还没有达到生产级系统的成熟度，更像是“单机演示型原型”。主要问题集中在：分层边界不清、全局状态较多、领域模型和接口模型混用、任务执行同步阻塞、存储不可恢复、基础设施能力和业务逻辑耦合、异常治理不足、扩展点不稳定、缺少配置体系和工程质量门禁。

## 当前架构简述

当前代码大致可以分为以下模块：

- `app/main.py`：FastAPI 应用入口、路由定义、全局服务对象初始化、环境和账号配置。
- `app/services.py`：文档服务、解析任务服务、执行任务服务、日志服务。
- `app/runner.py`：Playwright 执行器，负责步骤执行、断言执行、浏览器生命周期和证据采集。
- `app/llm_agent.py`：OpenAI-compatible LLM 调用、用例解析、步骤规划和断言规划。
- `app/store.py`：内存存储。
- `app/models.py`：Pydantic 模型、枚举、请求模型、响应模型、领域模型混合定义。
- `app/evidence.py`：证据文件写入和附件路径管理。
- `app/reporting.py`：Allure 风格 JSON 和 HTML 报告生成。

这个结构适合快速验证想法，但随着需求增长，会明显遇到维护、扩展、并发、部署和稳定性瓶颈。

## 主要架构不足

### 1. 应用入口承担了过多职责

`app/main.py` 同时负责：

- FastAPI app 创建。
- 路由注册。
- 全局依赖对象初始化。
- 内存环境配置管理。
- 账号配置管理。
- 静态资源目录创建和挂载。
- 异常处理。

这会让入口文件逐渐变成“总控脚本”，不利于测试、替换实现和拆分模块。

建议：

- 拆出 `api/routers`，按资源划分路由：documents、parse_tasks、execution_tasks、config、artifacts。
- 拆出 `core/container.py` 或 `dependencies.py` 统一管理依赖注入。
- 拆出 `core/settings.py` 管理配置。
- 将环境和账号配置从 `main.py` 移到独立 service/repository。

### 2. 分层边界不够清晰

当前 `services.py` 同时包含多个业务服务，既做业务编排，也直接依赖具体存储、文件证据、报告生成和 Playwright Runner。

问题：

- 服务层直接知道太多基础设施细节。
- 后续要替换数据库、任务队列、报告引擎、浏览器执行器时，改动面会扩大。
- 一个文件承载多个 bounded context，复杂度会继续堆积。

建议按职责拆分：

- `domain/`：领域模型、领域服务、状态机、领域事件。
- `application/`：用例编排服务，例如创建解析任务、创建执行任务。
- `infrastructure/`：数据库、文件存储、LLM client、Playwright runner、报告生成。
- `api/`：HTTP DTO、路由、异常映射。

### 3. 领域模型、请求模型、响应模型混用

`app/models.py` 里同时定义了：

- API 请求模型，例如 `CreateExecutionTaskRequest`。
- API 响应模型，例如 `UploadDocumentResponse`、`CreateTaskResponse`。
- 领域实体，例如 `ExecutionTask`、`ParsedTestCase`、`TaskLog`。
- 运行结果模型，例如 `StepExecutionResult`、`AssertionResult`。

这会带来两个长期问题：

- API 字段变化容易污染领域模型。
- 内部领域字段容易无意暴露给外部接口。

建议：

- 拆分 `schemas/` 和 `domain/models.py`。
- 外部 API 使用独立 request/response DTO。
- 领域层只表达业务概念，不绑定 HTTP 展示。
- 在应用层做 DTO 和领域对象转换。

### 4. 全局可变状态过多

当前 `main.py` 中存在全局对象：

- `store = InMemoryAgentStore()`
- `documents_service`
- `parse_tasks_service`
- `execution_tasks_service`
- `environments: list[dict[str, Any]]`
- `accounts: list[dict[str, Any]]`

风险：

- 多 worker 部署时状态不共享。
- 服务重启后任务、配置、账号全部丢失。
- 并发请求下可能出现数据竞争。
- 测试之间容易互相污染。

建议：

- 用数据库替代内存状态。
- 用 FastAPI lifespan 初始化资源。
- 通过依赖注入提供 service/repository。
- 环境配置、账号配置进入持久化配置表。

### 5. 存储抽象不足

`InMemoryAgentStore` 是具体实现，但服务层直接依赖它，而不是依赖抽象接口。

问题：

- 后续替换成 SQLite、PostgreSQL、Redis 或对象存储时，服务层需要改动。
- 缺少事务边界，任务、用例、日志、报告状态无法原子更新。
- 缺少查询能力，例如分页、按状态筛选、按创建时间排序。

建议：

- 定义 repository protocol/interface：`DocumentRepository`、`ParseTaskRepository`、`ExecutionTaskRepository`、`TaskLogRepository`。
- 服务层依赖接口，不依赖具体实现。
- 基础设施层实现内存版和数据库版。
- 对任务状态更新使用事务。

### 6. 缺少明确的任务状态机

当前任务状态是直接赋值，例如 parsing、running、finished、failed，但没有集中状态机约束。

风险：

- 可能出现非法状态流转。
- 异常发生时任务状态可能停留在 running/parsing。
- 未来支持取消、重试、暂停、恢复时会变复杂。

建议：

- 建立任务状态机模块。
- 明确定义允许流转：created -> parsing/running -> finished/failed/canceled。
- 所有状态变化通过统一方法完成。
- 状态变化同时记录事件日志和时间戳。

### 7. 同步执行架构不适合长任务

`ExecutionTaskService.create()` 中直接同步执行所有用例：

```python
results = [self._run_case(task, test_case, context) for test_case in cases]
```

这对 UI 自动化是不合适的，因为浏览器执行可能持续数分钟甚至更久。

问题：

- HTTP 请求容易超时。
- 单个请求长期占用应用线程。
- 不支持任务取消、排队、限流、并发执行。
- 无法横向扩容执行能力。

建议：

- API 只创建任务并返回 taskId。
- 执行交给 worker 或后台任务。
- 引入任务队列，例如 Celery、RQ、Dramatiq，或先使用 FastAPI BackgroundTasks 做过渡。
- 设计执行器并发池和浏览器资源配额。

### 8. LLM 基础设施和领域逻辑耦合较重

`OpenAICompatibleLLMAgent` 同时承担：

- prompt 拼接。
- HTTP 请求。
- JSON 解析。
- schema 校验。
- 领域对象 `ParsedTestCase` 构造。

问题：

- Prompt 难以版本化和灰度。
- LLM Provider 难以替换。
- 网络异常、模型异常、schema 异常缺少统一治理。
- LLM 审计数据没有沉淀。

建议：

- 拆分 `PromptTemplate`、`LLMClient`、`LLMResponseParser`、`CasePlanningService`。
- 将 prompt 模板独立为可版本管理的资源。
- 记录 model、prompt version、request id、response hash、耗时和错误类型。
- LLM 返回先转换为 plan DTO，再由应用层转换为领域对象。

### 9. Runner 职责过重

`PlaywrightExecutionRunner` 目前负责：

- 浏览器生命周期。
- 页面事件监听。
- 步骤规划调用。
- 动作执行。
- 断言执行。
- URL 校验。
- Locator 解析。
- 敏感信息脱敏。
- 截图和 Trace 采集。
- Console/Network 证据写入。
- 日志记录。

这是典型的“上帝类”趋势。

建议拆分：

- `BrowserSessionManager`：浏览器和上下文生命周期。
- `StepPlanner`：步骤到动作计划。
- `ActionExecutor`：动作执行。
- `AssertionExecutor`：断言执行。
- `LocatorResolver`：定位器策略。
- `EvidenceRecorder`：证据采集。
- `SensitiveDataMasker`：脱敏。

这样后续扩展 iframe、弹窗、文件上传、自愈定位、视频录制时不会全部塞进一个类。

### 10. 配置体系缺失

当前配置来源比较分散：

- LLM 使用环境变量。
- 数据目录写死默认 `data`。
- 默认环境配置写在 `main.py`。
- 浏览器和执行参数来自请求。
- 静态资源路径在入口中创建。

建议：

- 使用 Pydantic Settings 建立统一配置。
- 配置项覆盖：服务端口、数据目录、数据库、LLM、浏览器默认值、任务并发、文件大小限制、报告保留周期。
- 区分 dev/test/prod 环境。
- 不在业务代码里硬编码默认生产行为。

### 11. 错误处理和异常边界不完整

当前只对 `BadRequestError` 和 `ResourceNotFoundError` 做了统一映射。LLM、文件 IO、Playwright、JSON 生成、Unicode decode 等异常没有被应用层统一兜底。

风险：

- 内部异常直接变成 500。
- 任务失败时状态可能没有落库。
- 用户只能看到技术异常，无法定位业务原因。

建议：

- 建立统一错误码体系。
- 应用层捕获基础设施异常并转换为领域失败。
- 执行任务异常时确保任务进入 failed 状态。
- 日志中记录完整技术细节，API 中返回安全可读信息。

### 12. 文件存储和静态资源暴露设计偏简单

证据、报告和文档都放在本地 `data` 目录，静态挂载直接暴露 `/artifacts` 和 `/reports/allure`。

问题：

- 没有访问控制。
- 文件路径和 URL 生成规则散落在代码中。
- 本地磁盘不适合多实例部署。
- 缺少生命周期管理和清理策略。

建议：

- 抽象 `ArtifactStorage` 接口，支持 local、S3、MinIO。
- 文件访问通过受控下载接口或短期签名 URL。
- 增加清理任务和保留策略。
- 报告和证据元数据进入数据库。

### 13. 安全架构薄弱

当前架构没有鉴权、权限、租户隔离和敏感信息保护的整体设计。

风险：

- 任意用户可创建环境、账号、执行任务。
- 报告和截图可能暴露业务数据。
- 账号虽然返回时脱敏，但实际存储在内存中没有加密能力。
- LLM 页面快照可能携带敏感信息。

建议：

- 增加认证和授权中间件。
- 资源模型增加 projectId/userId。
- 账号和密钥使用加密存储。
- 对 LLM 输入、日志、报告、证据做统一脱敏。
- 引入审计日志。

### 14. API 组织和契约治理不足

当前所有路由集中在 `main.py`，且部分接口直接返回内部对象。

建议：

- 按资源拆分 router。
- 所有接口声明 response_model。
- 对 `dict[str, Any]` 类型的配置请求建立强 schema。
- 补充 OpenAPI 示例和错误响应。
- 建立 API 版本兼容策略。

### 15. 缺少领域事件和可观测性设计

当前日志是任务日志，更多服务于业务查询，不等同于系统可观测性。

建议：

- 引入领域事件：DocumentUploaded、ParseTaskCreated、CaseReviewed、ExecutionTaskStarted、CaseFinished、ReportGenerated。
- 事件驱动日志、通知、指标、报告生成。
- 增加 metrics：任务耗时、执行成功率、LLM 调用成功率、定位失败率、报告生成失败率。
- 增加 trace/span，贯穿一次任务全链路。

### 16. 日志输出能力存在明显缺陷

当前项目已经有 `TaskLogService` 和 `/api/v1/execution-tasks/{task_id}/logs` 查询接口，说明已经意识到执行过程需要可追踪，这是一个好的起点。但从高级研发和生产排障角度看，现有日志体系仍然偏弱，不能只看“能不能查到日志”，还要看“出了问题能不能快速定位、能不能跨模块串联、能不能被平台采集、能不能审计和告警”。

当前主要问题：

- 日志只写入 `InMemoryAgentStore`，服务重启后全部丢失。
- 日志没有接入 Python 标准 `logging`，也没有输出到控制台、文件或外部日志平台。
- 业务任务日志和应用运行日志混在概念上，但实现上又完全割裂，Uvicorn/FastAPI 的访问日志、异常日志和任务日志无法统一检索。
- `traceId` 使用不统一，很多地方直接传 `taskId` 或 `parseTaskId`，无法表达一次完整链路中的 document、parse、execution、case、step、LLM 调用关系。
- 缺少 requestId、userId、projectId、environmentCode、documentId、parseTaskId、caseId、stepIndex 等稳定上下文字段。
- 日志事件枚举没有集中定义，`DOCUMENT_UPLOADED`、`PARSE_TASK_CREATED`、`CASE_EXECUTION_FINISHED` 等字符串散落在代码里，后续容易拼写不一致。
- 日志等级使用偏粗，很多失败路径只记录 `WARN`，缺少明确的 `ERROR` 和异常堆栈。
- LLM 调用没有日志记录，包括模型、耗时、prompt version、响应状态、错误类型、重试次数都缺失。
- Playwright 执行细节日志不足，浏览器启动、页面跳转、定位器解析、断言失败、截图失败、Trace 失败等关键环节没有充分记录。
- 证据写入异常被吞掉，例如截图和 Trace 采集失败时直接 `pass`，排查时不知道证据为什么缺失。
- 日志没有分页，任务日志多了以后查询接口会一次性返回全部数据。
- 缺少日志保留策略、归档策略和敏感信息治理策略。
- 日志文案存在中文乱码问题，会直接降低排障效率和交付专业度。

现有日志方便的地方：

- 可以按 `task_id` 查询任务内日志。
- 可以按 `caseRunId`、`stepRunId`、`level`、`event` 做基础过滤。
- 日志结构中有 `attributes`，具备扩展结构化字段的基础。

但这些能力还不足以支撑生产环境排障。比如一次执行失败后，研发通常需要快速回答：请求是谁发起的、用了哪个环境、解析结果是什么、LLM 规划了什么、浏览器执行到了哪一步、定位器是什么、页面状态是什么、截图和 Trace 是否生成、失败属于环境问题还是脚本问题。当前日志链路还回答不完整。

建议改造：

- 建立统一日志模块，例如 `core/logging.py`，统一配置 Python 标准 logging。
- 将任务日志、应用日志、访问日志统一使用结构化 JSON 输出。
- 定义统一上下文：`traceId`、`requestId`、`taskId`、`caseRunId`、`stepRunId`、`documentId`、`parseTaskId`、`environmentCode`。
- 使用 contextvars 或中间件自动注入 requestId/traceId。
- 建立日志事件枚举，例如 `LogEvent`，避免事件名散落在代码里。
- 对 LLM 调用增加专门日志：模型、耗时、成功/失败、错误类型、prompt version、响应摘要。
- 对 Playwright 关键动作增加步骤级日志：action、target、timeout、durationMs、status、errorType。
- 证据采集失败不能静默吞掉，至少记录 WARN，并写入 case summary。
- 将任务日志持久化到数据库，应用日志输出到 stdout 或文件，便于容器和日志平台采集。
- 增加日志分页和时间范围过滤。
- 对日志 attributes 做统一脱敏，避免 password、token、手机号、cookie 泄露。

推荐的日志分层：

| 日志类型 | 作用 | 建议落点 |
| --- | --- | --- |
| 应用运行日志 | 服务启动、依赖初始化、未捕获异常 | stdout / 日志平台 |
| HTTP 访问日志 | 请求路径、状态码、耗时、requestId | stdout / 网关 / APM |
| 任务事件日志 | 文档上传、解析、确认、执行、报告生成 | 数据库 |
| 步骤执行日志 | 每一步 action、target、结果、耗时 | 数据库 + 报告 |
| LLM 调用日志 | 模型调用、耗时、错误、响应摘要 | 数据库或审计表 |
| 证据日志 | 截图、Trace、Console、Network 文件状态 | 数据库 + artifact 元数据 |
| 审计日志 | 用户、账号、环境配置、任务操作 | 数据库，不允许随意删除 |

推荐的结构化日志字段：

```json
{
  "timestamp": "2026-05-13T10:00:00+08:00",
  "level": "INFO",
  "event": "STEP_EXECUTION_FINISHED",
  "traceId": "trace_xxx",
  "requestId": "req_xxx",
  "taskId": "exec_xxx",
  "caseRunId": "case_run_xxx",
  "stepRunId": "step_run_xxx",
  "environmentCode": "test",
  "action": "click",
  "target": "text=提交订单",
  "status": "passed",
  "durationMs": 531,
  "message": "Step execution finished"
}
```

### 17. 可测试性仍不够理想

虽然部分构造函数支持注入 fake runner 和 fake LLM，但整体仍存在可测试性短板。

问题：

- `main.py` 全局初始化不利于隔离测试。
- Runner 大量依赖 Playwright 运行时，缺少动作执行抽象。
- 文件系统写入没有统一临时目录注入机制。
- 时间、ID、配置都没有完全抽象。

建议：

- 提供 app factory：`create_app(settings, container)`。
- 抽象 `Clock`、`IdGenerator`、`Storage`。
- Runner 内部拆分后分别做单元测试。
- API 测试使用独立容器和临时存储。

### 18. 工程结构不利于持续演进

当前 `app` 目录扁平，随着功能增长会快速膨胀。

建议演进为：

```text
app/
  api/
    routers/
    schemas/
    exception_handlers.py
  application/
    document_service.py
    parse_task_service.py
    execution_task_service.py
  domain/
    models.py
    events.py
    state_machine.py
    policies.py
  infrastructure/
    repositories/
    llm/
    browser/
    storage/
    reporting/
  core/
    settings.py
    container.py
    logging.py
```

这不是为了“分目录而分目录”，而是为了让依赖方向变清楚：API 依赖应用层，应用层依赖领域抽象，基础设施实现抽象。

## 优先级建议

### P0：先解决架构风险最大的点

1. 修复中文编码和异常字符串问题，否则日志、接口和文档可信度都会受影响。
2. 拆分 `main.py` 路由和依赖初始化，建立 app factory。
3. 把 `models.py` 拆成 API schemas 和 domain models。
4. 为存储层定义 repository 抽象，不再让服务层直接依赖 `InMemoryAgentStore`。
5. 统一异常处理，让 LLM、Playwright、文件 IO 错误都能转成可诊断的任务失败。
6. 建立统一日志规范，至少补齐 traceId/requestId、结构化日志、关键失败路径日志和证据采集失败日志。

### P1：支撑可试用和多人协作

1. 引入数据库持久化。
2. 执行任务异步化，建立任务队列或 worker。
3. 建立任务状态机。
4. 拆分 Runner，形成动作执行、断言执行、证据采集、浏览器会话等独立组件。
5. 建立统一配置体系。
6. 将任务日志持久化，并增加日志分页、时间过滤、事件枚举和敏感信息脱敏。

### P2：走向生产化

1. 增加鉴权、权限、项目隔离和审计。
2. 抽象文件存储，支持对象存储和受控访问。
3. 建立领域事件和指标监控。
4. 完善 CI、类型检查、覆盖率、契约测试和安全扫描。
5. 对 LLM prompt、模型版本、响应结果建立审计和回放能力。
6. 接入日志平台或 APM，支持跨请求、跨任务、跨执行步骤的链路追踪。

## 推荐的近期重构路径

第一阶段：不改变功能，只整理边界。

- 新增 `create_app()`。
- 拆路由到 `api/routers`。
- 拆 DTO 到 `api/schemas`。
- 拆 settings 到 `core/settings.py`。
- 保留内存存储，但通过 repository protocol 注入。

第二阶段：稳定执行链路。

- 增加任务状态机。
- 执行任务从同步改为异步。
- 统一异常转换。
- Runner 拆成多个小组件。

第三阶段：生产化基础设施。

- 数据库存储替代内存存储。
- artifact storage 抽象化。
- 增加认证授权和敏感信息治理。
- 增加可观测性和质量门禁。

## 总体判断

当前代码架构适合 MVP 验证，但还不适合作为长期演进的生产级架构。它现在的核心价值是把业务链路跑通了；下一步的重点应该是把“能跑”升级为“可维护、可扩展、可恢复、可观测、可替换”。

如果以高级研发工程师标准推进，建议优先做架构边界重构和任务执行模型重构，而不是继续在现有文件里堆更多功能。
