# 测试工程师经验水准评估不足清单

评估日期：2026-05-13

## 结论

从当前项目形态看，项目已经具备 UI 自动化测试 Agent 的 MVP 闭环：Markdown 用例上传、LLM 解析、人工确认、Playwright 执行、证据采集、报告生成和基础接口查询都已经搭起来了。

但如果站在 10 年测试工程师的交付标准来看，当前项目还没有达到成熟测试工程平台或生产级测试基础设施的程度。主要差距不在“有没有功能雏形”，而在稳定性、可测性、覆盖深度、工程治理、安全边界、异常恢复、质量度量和持续集成闭环上。

当前测试执行结果：`.venv\Scripts\python -m pytest -q`，2 个测试通过。

## 已具备的基础能力

1. 具备 FastAPI 后端接口，核心资源路径较完整。
2. 具备 Markdown 用例解析、解析确认、执行任务创建、日志查询、报告查询等主流程。
3. 已接入 Playwright，具备真实浏览器执行的基础框架。
4. 已有证据输出，包括截图、Trace、Console、Network、summary JSON。
5. 已有简单 HTML 报告和 Allure 风格结果文件。
6. 已有少量单元测试，能验证解析代理调用和执行服务的基础成功路径。

## 主要不足

### 1. 测试策略不完整

当前只有 2 个测试用例，覆盖的是非常理想化的 happy path。缺少完整的测试金字塔设计，也没有明确区分单元测试、接口测试、集成测试、端到端测试、契约测试和回归测试。

建议补齐：

- 服务层单元测试：文档上传、解析任务、确认流程、执行任务、日志过滤、报告生成。
- API 测试：所有 REST 接口的成功、失败、参数校验、资源不存在、非法状态流转。
- Runner 集成测试：使用可控测试页面验证 goto、click、fill、select、wait、scroll、press 和断言行为。
- 端到端测试：从上传 Markdown 到生成报告的完整闭环。
- 回归测试集：覆盖历史缺陷和高风险路径。

### 2. 异常路径覆盖不足

目前测试没有覆盖以下高频失败场景：

- 上传空文件、非 `.md` 文件、非 UTF-8 文件。
- LLM 未配置 API Key。
- LLM 返回非 JSON、结构不匹配、超时、HTTP 错误。
- Playwright 未安装、浏览器启动失败、页面超时、元素定位失败。
- `documentId` 和 `parseTaskId` 不匹配。
- 解析任务不存在、执行任务不存在、环境配置不存在。
- 无可执行用例、用例被跳过、用例 invalid、用例 need_review。
- 报告目录或证据目录不可写。

10 年经验的测试工程师通常会优先补齐这些失败路径，因为它们决定系统在真实环境中的可诊断性和抗故障能力。

### 3. 编码和文档质量存在明显问题

README、接口返回文案和源码中的中文字符串存在乱码现象。这会影响 API 响应可读性、用户文档可信度、日志排查效率和交付专业度。

建议：

- 统一项目文件编码为 UTF-8。
- 修复 README、Apifox 操作指南、需求文档和源码中的乱码。
- 在 CI 中增加编码检查或文档渲染检查。
- 对 API 返回文案做统一 i18n 或常量管理。

### 4. 依赖和配置存在不一致

`pyproject.toml` 的依赖中没有声明 `playwright`，但 `requirements.txt` 中声明了 `playwright`。这会导致不同安装方式下运行结果不一致。

建议：

- 统一依赖来源，明确以 `pyproject.toml` 还是 `requirements.txt` 为准。
- 增加 lock 文件或固定关键依赖版本。
- 明确 Playwright 浏览器安装步骤，并在启动或健康检查中提示缺失依赖。

### 5. 持久化能力不足

当前核心数据使用 `InMemoryAgentStore`，服务重启后任务、日志、解析结果、执行结果都会丢失。虽然证据文件写入本地磁盘，但元数据不具备可靠恢复能力。

建议：

- 引入数据库存储任务、用例、日志、执行结果和环境配置。
- 增加任务状态机，避免非法状态流转。
- 增加数据迁移、备份、清理策略。
- 明确任务、报告、证据文件的保留周期。

### 6. 执行任务是同步阻塞模式

`create_execution_task` 当前会直接同步执行全部用例。真实 UI 自动化执行通常耗时较长，同步接口容易导致请求超时、服务阻塞和并发能力不足。

建议：

- 将解析和执行改为异步任务。
- 引入任务队列或后台 worker。
- 支持任务取消、暂停、重试、超时控制和并发限制。
- 接口只负责创建任务，执行进度通过查询接口或事件推送获取。

### 7. LLM 行为不可控，缺少质量护栏

项目把 Markdown 解析、步骤规划和断言规划都交给 LLM，但当前缺少充分的校验、回退和可解释机制。

主要风险：

- LLM 输出不稳定，导致同一用例多次执行结果不同。
- 步骤规划可能生成危险 URL 或错误定位器。
- 断言可能过宽或过窄，造成误报和漏报。
- 没有记录 prompt、模型、响应原文和 token 信息，问题难以复现。

建议：

- 保存每次 LLM 请求和响应的审计记录，敏感信息脱敏。
- 对 LLM 输出做二次规则校验。
- 建立确定性解析器兜底，优先处理规范 Markdown。
- 为 LLM 输出建立 golden case 测试集。
- 增加模型失败后的重试、降级和人工复核机制。

### 8. 自动化执行能力仍偏基础

当前 Runner 支持的动作和断言有限，尚未达到复杂业务 UI 自动化平台的能力。

不足包括：

- 缺少 iframe、弹窗、新窗口、文件上传、拖拽、悬浮、下载、验证码绕过策略等支持。
- 缺少 locator 自愈的真实实现，虽然状态中已有 `locator_healed`。
- 缺少步骤级截图、视频、DOM 快照、失败前后页面状态对比。
- 缺少网络请求 mock、接口等待、接口断言和业务数据校验。
- 缺少测试数据准备和清理机制。

### 9. 安全边界不够完善

项目已有 domain whitelist，但整体安全控制仍不足。

建议重点补齐：

- 环境配置和账号配置需要鉴权。
- 密码不应只返回时脱敏，存储也不能明文。
- 上传文件需要大小限制、内容校验和恶意内容处理。
- 证据和报告静态资源需要访问控制，避免泄露截图、token、手机号等敏感信息。
- LLM prompt 和页面快照中需要更严格的敏感信息脱敏。
- 外部 URL 访问需要更细粒度限制，防止 SSRF 类风险。

### 10. API 质量和契约治理不足

当前接口能工作，但生产级接口治理还不够。

不足包括：

- 部分接口直接返回内部模型，缺少稳定的 response DTO。
- 环境和账号配置使用裸 `dict[str, Any]`，缺少 schema 校验。
- 错误码体系较简单，没有统一的业务错误分类。
- 缺少分页、排序、筛选的标准化。
- 缺少 OpenAPI 示例、字段说明和兼容性策略。
- 缺少接口契约测试。

### 11. 日志和可观测性不足

当前有任务日志，但还不是完整的可观测体系。

建议补齐：

- 统一 traceId，并贯穿文档、解析、执行、用例、步骤、LLM 调用和报告生成。
- 增加结构化应用日志。
- 增加 metrics：执行耗时、成功率、失败类型分布、LLM 成功率、定位失败率。
- 增加健康检查细分项：数据库、磁盘、Playwright、LLM 连接、报告目录。
- 增加告警策略。

### 12. 报告能力偏简化

当前 HTML 报告只展示用例行级结果，Allure 结果也是基础 JSON，并未完整发挥测试报告的定位价值。

建议：

- 报告中展示步骤详情、断言详情、截图、Trace、Console、Network 链接。
- 按模块、优先级、标签、失败原因聚合。
- 增加趋势统计和历史对比。
- 增加失败归因：环境问题、脚本问题、断言问题、业务缺陷、数据问题。
- 对 Allure 结果结构做兼容性验证。

### 13. 工程质量门禁不足

当前没有看到 CI、覆盖率、静态检查、格式化、类型检查、安全扫描等质量门禁。

建议：

- 增加 CI 流水线，至少运行 pytest、ruff、mypy 或 pyright。
- 增加测试覆盖率统计和阈值。
- 增加依赖漏洞扫描。
- 增加 pre-commit。
- 增加构建、启动、接口 smoke test。

### 14. 并发和线程安全风险

`InMemoryAgentStore` 使用普通 dict/list，没有并发保护。FastAPI 在并发请求下可能出现数据竞争、状态覆盖和日志顺序异常。

建议：

- 引入真实数据库或线程安全存储层。
- 对任务创建、状态更新和日志写入做事务化处理。
- 增加并发测试和压力测试。

### 15. 测试数据管理缺失

当前用例执行没有完整的数据准备、账号管理、数据隔离和清理机制。

建议：

- 支持按环境配置测试账号、测试数据和前置脚本。
- 支持执行前 setup、执行后 teardown。
- 支持数据唯一化，避免多次运行相互污染。
- 对账号和密钥做安全存储。

### 16. 产品化体验不足

项目更像后端能力验证，还不是完整的测试平台产品。

不足包括：

- 缺少前端管理界面。
- 缺少任务进度实时展示。
- 缺少失败用例重跑。
- 缺少用例编辑、版本管理、审批流。
- 缺少团队协作、权限、审计和项目隔离。

## 优先级建议

### P0：必须尽快处理

1. 修复中文乱码和文档编码问题。
2. 补齐异常路径测试和 API 测试。
3. 统一依赖声明，确保 Playwright 安装路径清晰。
4. 给环境配置、账号配置、上传文件和静态报告增加安全边界。
5. 将 LLM 调用失败转化为可控业务错误，而不是让任务不可诊断地失败。

### P1：进入可试用阶段前处理

1. 引入持久化存储。
2. 执行任务异步化。
3. 增加任务状态机、取消、重试和超时控制。
4. 增加 Runner 集成测试和端到端测试。
5. 完善报告详情和证据链。

### P2：走向成熟平台时处理

1. 建立 CI/CD 质量门禁。
2. 增加趋势分析、失败归因和质量度量。
3. 增加 locator 自愈、复杂交互和数据管理能力。
4. 增加权限、审计、多项目隔离和团队协作能力。

## 建议的测试补充清单

| 类型 | 建议用例 |
| --- | --- |
| 文档上传 | 空文件、非 md 文件、大文件、中文文件名、非 UTF-8 内容 |
| 解析任务 | 正常解析、LLM 未配置、LLM 返回非法 JSON、空用例、need_review 用例 |
| 人工确认 | confirm、revise、skip、mark_invalid、caseId 不存在、空列表 |
| 执行任务 | 环境不存在、documentId 不匹配、无可执行用例、筛选优先级、筛选标签 |
| Runner | goto、click、fill、select、wait、scroll、press、blocked、locator 失败 |
| 断言 | text_visible、url_contains、no_console_error、element_enabled、element_disabled |
| 报告 | 成功报告、失败报告、附件链接、Allure JSON 结构 |
| 安全 | 越权访问、敏感信息脱敏、非法 URL、文件上传限制 |
| 并发 | 多任务同时创建、日志顺序、任务结果隔离 |

## 总体判断

当前项目可以算作“有工程意识的 MVP”或“可继续演进的原型”，但还不能算达到 10 年测试工程师应交付的成熟水准。

如果要接近 10 年经验测试工程师的标准，下一步不应只继续堆功能，而应优先补齐质量工程能力：测试覆盖、异常治理、稳定执行、可观测性、安全边界、持续集成和可复现的质量度量。
